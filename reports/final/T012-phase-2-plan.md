# T012 第二阶段任务路线规划报告

## 1. 规划背景

第一阶段已完成最小自动化闭环，runner.py 能自动读取任务、生成提示词、调用 Claude Code、判断结果、更新状态。

第二阶段在此基础上扩展自动化能力：协议规范化、接入国内模型、智能 Agent、真实项目验证。

## 2. 第二阶段核心目标

1. **协议规范化** — 工作流、Agent 输出、需求输入
2. **模型接入** — 国内模型 API 适配器 MVP
3. **智能 Agent** — Planner 自动拆解、Reviewer 自动审查
4. **真实验证** — 用小游戏项目验证端到端流程

## 3. 任务路线（T013-T022，共 10 个任务）

| 任务 | 名称 | 方向 | 依赖 | 复杂度 |
|------|------|------|------|--------|
| T013 | 工作流协议规范化 | 协议 | 无 | 中 |
| T014 | Agent 输出协议规范化 | 协议 | 无 | 中 |
| T015 | 项目需求输入规范化 | 协议 | 无 | 低 |
| T016 | 国内模型 API 适配器 MVP | 模型 | 无 | 中高 |
| T017 | Planner Agent 自动拆解 MVP | Agent | T015, T016 | 高 |
| T018 | Main Agent 决策协议 MVP | 调度 | T014 | 中 |
| T019 | 小游戏验证项目初始化 | 验证 | T015 | 低 |
| T020 | 自动执行小游戏开发任务 | 验证 | T019 | 中 |
| T021 | Reviewer Agent 自动审查 MVP | Agent | T014, T016 | 高 |
| T022 | 第二阶段阶段总结 | 总结 | 全部 | 低 |

## 4. 推荐执行顺序

```
T013/T014/T015 → T016 → T017 → T018 → T019 → T020 → T021 → T022
```

阶段划分：

- **协议设计阶段（T013-T015）：** 统一工作流、输出、输入格式
- **模型接入阶段（T016）：** 打通国内模型 API
- **智能 Agent 阶段（T017-T018）：** Planner 拆解、Main Agent 决策
- **验证阶段（T019-T020）：** 小游戏项目端到端验证
- **质量与总结（T021-T022）：** Reviewer 审查、阶段总结

## 5. 暂不做的事情

- 不做 Web UI
- 不做数据库
- 不做多模型支持（先一个）
- 不做完整游戏
- 不做 Docker / 部署
- 不做并发执行

## 6. 关键风险

1. **模型 API 稳定性** → 适配器做好错误处理和重试
2. **Planner 生成质量** → 保留人工审核环节
3. **协议设计过度** → 够用即可，后续迭代
4. **验证项目范围膨胀** → 需求要克制
5. **兼容性** → 不破坏第一阶段流程

## 7. 预期成果

第二阶段完成后，框架应具备：

- 统一的协议规范（工作流、Agent 输出、需求输入）
- 国内模型 API 调用能力
- Planner Agent 自动任务拆解
- Reviewer Agent 自动审查
- Main Agent 决策规则
- 一个真实项目验证端到端流程

## 8. 与最终目标的关系

```
最终目标：用户提需求 → 自动拆解 → 自动开发 → 自动测试 → 自动审查 → 用户验收

第一阶段（已完成）：
  读取任务 → 生成 prompt → 调用 Claude Code → 判断结果 → 更新状态

第二阶段（本阶段）：
  + 协议规范（让多 Agent 协作有统一语言）
  + 模型接入（让非代码 Agent 有智能大脑）
  + Planner Agent（自动拆解需求）
  + Reviewer Agent（自动审查质量）
  + 真实项目验证（验证端到端）

第三阶段（未来）：
  + Tester Agent 自动测试
  + 返工闭环
  + 多项目支持
  + Web UI（可选）
```
