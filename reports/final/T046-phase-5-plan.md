## 1. 第五阶段定位

第五阶段是 `multi-agent-runner` 框架从"静态展示和简单交互"走向"物理逻辑自动化"的关键阶段。

前四阶段已经验证：
- runner 可以自动驱动子项目开发（G002-G005）
- Developer / Tester / Reviewer / Main Agent 四类证据闭环可行
- project.yaml 配置驱动、自动返工 prompt 生成、超时异常处理等安全机制已就位

第五阶段用 G006（简单重力下落）和 G007（玩家与平台基础碰撞）验证框架处理更复杂游戏逻辑的能力。

**第五阶段不是"快速做完整游戏"，而是继续用最小功能闭环验证框架的自动化能力。**

## 2. 第五阶段目标

1. 用 G006 验证框架可以驱动重力下落逻辑的自动开发和验证
2. 用 G007 验证框架可以驱动碰撞检测逻辑的自动开发和验证
3. 优先设计重力和碰撞的 Tester 协议，确保测试先于实现
4. 实现返工执行的人工确认版，从"只生成 prompt"走向"确认后执行"
5. 继续保持 Developer / Tester / Reviewer / Main Agent 完整证据链

## 6. 风险

1. 重力逻辑涉及时间帧和位置更新，Claude Code 生成逻辑可能需要人工确认
2. 碰撞检测涉及坐标重叠判断，静态测试可能无法完全覆盖运行时行为
3. Tester 静态检查对物理逻辑有局限性，后续可能需要浏览器自动化
4. 模型 429 / 超时风险仍然存在
5. 返工执行是第一次让框架真正执行返工，需要严格限制
6. script.js 复杂度增加，prompt 应明确已有代码上下文
