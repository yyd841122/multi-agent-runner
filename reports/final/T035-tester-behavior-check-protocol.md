# T035 Tester 行为检查协议报告

## 1. 背景

T029 定义了 Tester Agent 最小测试协议，T030 实现了本地静态检查 MVP。当前 Tester 可以检查文件存在、HTML 元素、CSS 样式和 JS 基础逻辑，共 16 项静态检查。

G003 和 G004 都已通过 Tester 静态检查并获得 PASS。

## 2. 当前问题

G004 实现了玩家键盘左右移动，Tester 返回 PASS（16/16 通过），但当前静态检查不能充分验证：

- 键盘事件是否真的被监听
- 左右方向键是否被正确识别
- 玩家移动是否有边界限制
- 位置更新函数是否被调用

G004 的基础静态测试 PASS 只说明页面结构和基础 JS 逻辑存在，不代表键盘移动交互行为被充分验证。

## 3. 协议目标

设计 Tester 行为检查协议，通过源码静态分析补充验证交互行为逻辑，为 T036 实现提供协议基础。

具体目标：

- 明确行为检查范围和检查项
- 定义键盘事件、移动逻辑、边界限制、位置更新四组检查项
- 定义 PASS / FAIL 规则
- 定义测试报告格式和完成证据路径
- 保持与现有 tester-protocol.md 的一致性

## 4. 行为检查范围

第一版行为检查基于源码静态检查，不引入浏览器自动化。

检查对象：

- `script.js` — 主要检查目标
- `index.html` — 辅助检查
- `style.css` — 辅助检查

四组检查项：

| 组 | 编号范围 | 检查内容 | 检查项数量 |
|---|---------|---------|-----------|
| 键盘事件 | B-01 ~ B-03 | 事件监听、方向键处理 | 3 |
| 移动逻辑 | M-01 ~ M-04 | 位置变量、左右移、步长 | 4 |
| 边界限制 | L-01 ~ L-03 | 左边界、右边界、越界防护 | 3 |
| 位置更新 | U-01 ~ U-03 | 更新函数、DOM 更新、调用链 | 3 |

共计 13 项行为检查。

## 5. 检查项设计

### 5.1 键盘事件检查（B 组）

| 编号 | 测试项 | 检查目标 |
|------|--------|----------|
| B-01 | 键盘事件监听存在 | `keydown` 或 `addEventListener('keydown'` 或 `onkeydown` |
| B-02 | 左方向键处理存在 | `ArrowLeft` 或 `keyCode === 37` |
| B-03 | 右方向键处理存在 | `ArrowRight` 或 `keyCode === 39` |

### 5.2 移动逻辑检查（M 组）

| 编号 | 测试项 | 检查目标 |
|------|--------|----------|
| M-01 | 玩家横向位置变量存在 | `playerState.x` 或 `playerX` 或 `player.x` |
| M-02 | 左移逻辑存在 | `x -` 或 `x -=` 或左移相关赋值 |
| M-03 | 右移逻辑存在 | `x +` 或 `x +=` 或右移相关赋值 |
| M-04 | 移动速度或步长存在 | `moveSpeed` 或 `speed` 或数值步长 |

### 5.3 边界限制检查（L 组）

| 编号 | 测试项 | 检查目标 |
|------|--------|----------|
| L-01 | 左边界限制存在 | `Math.max` 或 `>= 0` 或 `x < 0` |
| L-02 | 右边界限制存在 | `Math.min` 或 `areaWidth - playerWidth` |
| L-03 | 玩家不能移出游戏区域 | 同时存在左边界和右边界判断 |

### 5.4 位置更新检查（U 组）

| 编号 | 测试项 | 检查目标 |
|------|--------|----------|
| U-01 | 位置更新函数存在 | `updatePlayerPosition` 或 `updatePosition` |
| U-02 | DOM 位置更新存在 | `style.left` 或 `style.transform` |
| U-03 | 移动后调用更新函数 | 移动逻辑中调用位置更新函数 |

## 6. PASS / FAIL 规则

| 条件 | 结果 |
|------|------|
| 全部 13 项必需项通过 | `PASS` |
| 任一必需项失败 | `FAIL` |
| 源码文件缺失 | `BLOCKED` |
| 源码无法读取 | `BLOCKED` |

与基础静态检查的关系：

- 基础静态检查和行为检查独立判定
- 两者都 PASS 才能视为 Tester 整体 PASS
- Main Agent 应同时读取两个 Tester 证据

## 7. 完成证据路径

| 检查类型 | 完成证据 |
|----------|----------|
| 基础静态检查 | `<project>/reports/test/<task-id>-test-report.md` |
| 行为检查 | `<project>/reports/test/<task-id>-behavior-test-report.md` |

两个报告独立存在，便于 Main Agent 综合判断。

## 8. T036 实现建议

1. 在 `tools/tester_runner.py` 中新增 `run_behavior_check(project_path, task_id)` 函数
2. 读取 `script.js` 源码，按 B / M / L / U 四组检查项逐项搜索关键词
3. 生成行为检查报告到 `<project>/reports/test/<task-id>-behavior-test-report.md`
4. 不引入浏览器自动化，不做 AST 解析，保持简洁
5. 新增函数与已有静态检查函数分离，不破坏现有功能
