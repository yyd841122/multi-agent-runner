# T025.1 通用 Project Runner 首次成功总结

## 1. 背景

`multi-agent-runner` 第二阶段实现了 `run-game-next` 专用命令，成功驱动 `down-100-floors-game` 完成 G002。但该命令硬编码了项目路径和任务前缀，无法复用。

第三阶段 T024 设计了通用 project runner 协议，T025 实现了 `run-project-next` MVP。现在需要验证这个通用命令能否在真实场景中工作。

## 2. 本次目标

验证 `run-project-next` 能否成功驱动 `down-100-floors-game` 自动完成 G003（实现玩家角色显示）。

## 3. 执行命令

```powershell
python runner.py run-project-next --project projects/down-100-floors-game
```

## 6. 关键意义

这次验证证明 `multi-agent-runner` 已经从专用 `run-game-next` 迈向通用 `run-project-next`，可以基于项目路径驱动真实子项目继续自动开发。

具体意义：

1. **通用性验证** — `--project` 参数可以指向任意符合约定的子项目目录
2. **协议可行性** — T024 设计的协议（路径约定、任务解析、证据检查）在实际执行中有效
3. **自动化闭环** — 从读任务到标记 done 的完整闭环在通用模式下正常工作
4. **可扩展基础** — 未来新项目只需遵循目录约定，即可被 `run-project-next` 驱动

## 7. 成功链路

```
run-project-next --project <path>
    ↓
validate_project_root()          → 校验路径存在
    ↓
get_project_tasks_file()         → 定位 tasks.md
    ↓
parse_project_tasks()            → 通用解析（支持 G/T/P 前缀）
    ↓
find_next_pending_project_task() → 找到 G003
    ↓
update_project_task_status()     → 标记 in_progress
    ↓
build_project_task_prompt()      → 生成子项目 prompt
    ↓
run_claude_code()                → 调用 Claude Code
    ↓
get_project_dev_report_path()    → 检查完成证据
    ↓
update_project_task_status()     → 标记 done
```

## 8. 当前局限

1. **只验证了一个项目** — 当前只验证了 `down-100-floors-game`，尚未在不同项目类型中验证
2. **只支持单任务执行** — 没有多任务循环（相当于 `run-loop` 的项目版本）
3. **没有读取 project.yaml** — 当前只依赖路径约定，没有读取可选的项目配置文件
4. **没有文件权限校验** — prompt 中写了允许/禁止范围，但没有在代码层强制检查
5. **报告只保存到主项目** — Claude Code 输出保存到主项目 `reports/claude/`，子项目只有 dev-report
