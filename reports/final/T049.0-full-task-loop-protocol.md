# T049.0 单任务完整闭环自动化命令协议报告

## 1. 背景

当前系统已经具备各 Agent 的单步能力（Developer、Tester、Reviewer、Main Agent），但用户仍需手动依次执行多个命令才能完成一个任务的完整闭环。

单任务完整闭环是将半自动走向自动化落地的关键一步。

## 2. 当前问题

- 用户需要手动执行 4-5 个命令才能完成一个任务
- 手动执行容易遗漏步骤
- 无法快速验证完整闭环是否正常
- 返工流程更加复杂，需要更多手动操作

## 3. 目标命令

```powershell
python runner.py run-project-task-full --project projects/down-100-floors-game --task G006
```

一个命令完成单个任务的全生命周期。

## 4. 完整闭环流程

1. 读取 `project.yaml` 和子项目 `tasks.md`
2. 定位目标任务
3. 执行 Developer（如需要）
4. 检查 Developer 报告
5. 执行基础 Tester
6. 根据任务类型选择专项 Tester（如需要）
7. 执行 Reviewer
8. 执行 Main Agent 综合决策
9. 如果 COMPLETE，结束
10. 如果 REQUEST_CHANGES，生成 rework prompt
11. 如果超过最大返工次数，生成人工介入报告

## 5. 各阶段规则

### Developer

- pending → 执行 Developer
- done + 报告存在 → 跳过
- in_progress + 无报告 → 停止，提示确认
- 超时/失败 → 停止闭环
- completed_with_model_error → 可继续，但必须记录

### Tester

- Developer 报告存在后才执行
- FAIL → 不进入 Reviewer，生成 rework prompt
- BLOCKED → 停止并提示人工检查

### 专项 Tester

- G004 → keyboard behavior
- G006 → gravity
- G007 → collision
- 无映射 → 跳过

### Reviewer

- Tester PASS 后才执行
- APPROVE → 继续 Main Decision
- REQUEST_CHANGES → 生成 rework prompt
- API 429 / 调用失败 → BLOCKED

### Main Agent

- 读取 Developer / Tester / 专项 Tester / Reviewer 所有报告
- COMPLETE / REQUEST_CHANGES / BLOCKED

## 6. 失败处理

- 第一版 full loop 只生成 rework prompt，不自动执行返工
- 最大返工次数：3
- 超限后生成人工介入报告

## 7. 人工介入边界

以下情况必须停止并人工介入：

- Claude Code 超时
- Reviewer API 429
- DeepSeek API 调用失败
- Developer 报告缺失
- Tester BLOCKED
- Main Agent BLOCKED
- 超过最大返工次数
- 任务状态与证据冲突

## 8. 命令权限策略

### A 类（自动执行）

- 状态检查、文件读取、编译验证、测试、审查、决策命令

### B 类（仅 Git 备份任务）

- git add / commit / push

### C 类（需人工确认）

- 文件删除、恢复、移动

### D 类（禁止）

- git reset --hard、批量删除、rm -rf
