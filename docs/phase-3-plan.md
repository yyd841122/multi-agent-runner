# 第三阶段任务路线规划

## 1. 第三阶段目标

在第二阶段协议化和真实子项目验证基础上，向"更通用、更真实、更可验证"推进：

1. **通用项目执行器** — 将 `run-game-next` 泛化为可服务任意子项目的通用 project runner
2. **真实 Reviewer 模型** — 接入至少一个国内模型作为 Reviewer，替代 mock
3. **结构化审查解析** — 让 Reviewer 输出可被机器解析，支持自动决策
4. **Tester Agent 最小链路** — 建立本地静态检查能力，生成测试报告
5. **综合决策闭环** — Main Agent 综合 Developer / Tester / Reviewer 三方结果做决策
6. **继续小游戏验证** — 用 G003 继续验证框架自动化能力

**核心理念：第三阶段不是"做完整小游戏"，而是用小游戏继续验证和强化框架自动化能力。**

## 2. 前置基础

### 第一阶段（已完成）

- 最小自动化执行闭环：读任务 → 生成 prompt → 调用 Claude Code → 判断结果 → 更新状态
- 完成证据检查机制
- retry-current / run-next / run-loop 雏形
- 经验沉淀体系

### 第二阶段（已完成）

- Workflow / Agent Output / Requirement 三类协议
- 多模型 API 适配器 MVP（mock provider）
- Planner Agent 自动拆解链路
- Main Agent 决策协议（规则引擎）
- down-100-floors-game 验证项目（G001 / G002 已完成）
- Reviewer Agent 自动审查 MVP（mock provider）
- Git 备份策略

### 当前局限

1. `run-game-next` 只服务 down-100-floors-game，不可复用
2. Reviewer 只用 mock provider，无法生成真实审查结论
3. Reviewer 输出不可机器解析
4. 没有 Tester Agent，无自动化测试能力
5. Main Agent 决策只看 Developer 结果，不综合 Tester / Reviewer
6. 小游戏只有基础页面，没有实际玩法

## 3. 第三阶段设计原则

1. **通用化优先** — 先让框架能服务任意项目，再继续增加业务功能
2. **一个真实模型起步** — Reviewer 先接入一个国内模型，验证可用后再扩展
3. **最小测试起步** — 静态检查先行，不引入浏览器自动化
4. **保留回退能力** — mock provider 始终可用，真实模型只是可选增强
5. **保持 runner 稳定** — 新命令不破坏已有命令
6. **小任务验证** — 每个任务小而明确，可独立验证
7. **经验持续沉淀** — 每个任务完成后更新 memory

## 6. 暂不做的事情

1. **不做 Web UI 管理平台** — 保持 CLI 驱动
2. **不做数据库** — 继续使用文件系统
3. **不做微信小游戏 / 抖音小游戏发布** — 等 Web MVP 稳定后再适配
4. **不做角色技能系统** — 等基础玩法完成后再引入
5. **不做复杂浏览器自动化测试** — 先用静态检查
6. **不做多模型并发** — Reviewer 先接一个模型
7. **不做自动返工循环** — 综合决策先输出建议，不自动执行
8. **不做 Docker / 部署** — 专注核心自动化能力
9. **不做并发任务执行** — 保持串行
10. **不做用户认证 / 权限** — 单用户 CLI 工具

## 7. 风险与注意事项

1. **真实模型 API 稳定性** — 国内模型 API 可能有限额、超时、格式变更，适配器要做好错误处理和重试
2. **Reviewer 输出格式一致性** — 真实模型可能不严格遵循协议格式，需要解析容错
3. **通用 project runner 兼容性** — 泛化 run-game-next 时不能破坏 down-100-floors-game 已有功能
4. **测试报告质量** — 静态检查覆盖范围有限，不能替代真实运行测试
5. **综合决策复杂度** — 三方结果组合可能产生未预期的决策路径，规则要简洁明确
6. **小游戏功能范围控制** — G003 只做玩家显示和移动，不提前实现重力、碰撞等
