# 第五阶段任务路线规划

## 1. 第五阶段目标

第五阶段的核心目标是：

1. **继续用 G006 / G007 验证框架处理更复杂游戏逻辑的能力** — 重力下落和平台碰撞涉及物理逻辑，比前四阶段的静态展示和键盘移动更复杂。
2. **优先设计重力和碰撞的 Tester 协议** — 测试协议应先于实现，确保每个新能力都有对应的验证手段。
3. **实现返工执行的人工确认版** — 从"只生成返工 prompt"走向"用户确认后执行返工"，但不进入完全无人值守。

第五阶段不是"快速做完整游戏"，而是继续用最小功能闭环验证框架的自动化能力。

## 3. 第五阶段设计原则

1. **测试协议先于实现** — 先设计重力 / 碰撞的 Tester 协议，再实现对应游戏功能。
2. **最小可验证改进** — G006 只做简单重力下落，G007 只做基础碰撞，每个任务只增加一个能力。
3. **完整证据链** — 每个 G 任务仍保留 Developer / Tester / Reviewer / Main Agent 四类证据。
4. **安全返工** — 自动返工必须基于用户确认，不能直接无人值守循环。
5. **失败只记录** — 第五阶段如果 Tester / Reviewer / Main Agent 失败，只记录不自动返工。
6. **不扩大范围** — 不做完整游戏，不做平台滚动、随机平台、失败条件、角色技能、小游戏适配。

## 7. 风险与注意事项

1. **重力逻辑比显示和移动更复杂** — G006 涉及时间帧和位置更新，Claude Code 生成的逻辑可能需要人工确认。
2. **碰撞检测需要准确坐标判断** — G007 涉及玩家和平台的坐标重叠判断，静态测试可能无法完全覆盖。
3. **Tester 静态检查的局限性** — 重力和碰撞的运行时行为难以通过静态检查完全验证，后续可能需要引入浏览器自动化。
4. **模型 429 / 超时风险** — 前四阶段已遇到 429 和超时，第五阶段应继续使用 T043.0 / T043.1 的修复。
5. **返工执行需要谨慎** — T056 是第一次让框架执行返工，必须严格限制次数和范围。
6. **小游戏代码逐渐复杂** — G006 / G007 会增加 script.js 复杂度，prompt 中应明确已有代码上下文。
