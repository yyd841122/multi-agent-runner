# 第二阶段任务路线规划

## 1. 第二阶段目标

在第一阶段最小自动化闭环基础上，扩展自动化能力：

1. **协议规范化** — 定义工作流、Agent 输出、需求输入的统一格式
2. **接入国内模型** — 让 Planner、Reviewer 等 Agent 可以调用国内模型 API
3. **智能 Agent 实现** — 实现 Planner Agent 自动拆解、Reviewer Agent 自动审查
4. **真实项目验证** — 用小游戏项目验证框架能服务真实业务

## 2. 第一阶段基础

第一阶段已完成最小自动化闭环（T001-T011）：

- `runner.py` 支持 10 个命令（show / complete / start / generate-prompt / run-current / check-result / auto-complete-success / run-next / retry-current / run-loop）
- `tools/task_manager.py` — 任务解析和状态管理
- `tools/workflow_manager.py` — 提示词生成
- `tools/claude_code_runner.py` — Claude Code CLI 调用
- `tools/report_manager.py` — 报告生成和结果分析
- 6 个 Agent 角色定义（agents/*.md）
- 经验沉淀（memory/lessons.md、memory/pitfalls.md）

核心结论：

- runner.py 是最外层调度器
- Claude Code 是代码执行器
- 必须检查完成证据，不能只看退出码
- 自动化是最终目标

## 3. 第二阶段设计原则

1. **小步验证** — 每个任务小而明确，可独立验证
2. **先协议后实现** — 先定义格式规范，再写代码
3. **一个模型起步** — 国内模型只先支持一个，验证可用后再扩展
4. **验证项目驱动** — 用小游戏项目验证真实效果
5. **保持 runner 稳定** — 不重构第一阶段已验证的核心逻辑
6. **人工审核保留** — 关键环节（如任务拆解结果）需人工确认后才生效

## 4. 第二阶段任务路线

### T013 工作流协议规范化

**方向：** 协议规范化
**内容：**
- 定义 workflow YAML schema
- 包含：阶段名、Agent 角色、输入输出、执行顺序、验收规则
- 创建示例 workflow 文件
- 实现解析能力

**依赖：** 无
**复杂度：** 中

### T014 Agent 输出协议规范化

**方向：** 协议规范化
**内容：**
- 定义每个 Agent 角色的标准输出格式
- Developer → 修改文件列表 + 开发报告
- Tester → 测试结果 + 通过率
- Reviewer → 审查结论 + 建议
- Reporter → 总结报告
- Planner → 任务清单

**依赖：** 无（可与 T013 并行）
**复杂度：** 中

### T015 项目需求输入规范化

**方向：** 协议规范化
**内容：**
- 定义 requirement.md 标准格式
- 包含：项目名、目标、功能列表、技术约束、验收标准
- 实现解析函数

**依赖：** 无（可与 T013 / T014 并行）
**复杂度：** 低

### T016 国内模型 API 适配器 MVP

**方向：** 模型接入
**内容：**
- 创建 tools/model_adapter.py
- 定义统一调用接口
- 先支持一个国内模型（智谱或 DeepSeek）
- 支持错误处理和重试

**依赖：** 无
**复杂度：** 中高

### T017 Planner Agent 自动拆解任务 MVP

**方向：** 智能 Agent
**内容：**
- 读取需求文档
- 调用国内模型 API 生成任务清单
- 输出符合 tasks.md 格式
- 人工确认后写入

**依赖：** T015（需求格式）、T016（模型适配器）
**复杂度：** 高

### T018 Main Agent 决策协议 MVP

**方向：** 调度智能化
**内容：**
- 定义决策规则和输入输出
- 成功 / 失败 / 返工的不同处理路径
- 先用规则引擎，不接入 LLM

**依赖：** T014（Agent 输出协议）
**复杂度：** 中

### T019 down-100-floors-game 验证项目初始化

**方向：** 真实项目验证
**内容：**
- 创建项目目录结构
- 编写符合标准格式的需求文档
- 不实现游戏代码

**依赖：** T015（需求文档标准格式）
**复杂度：** 低

### T020 使用 runner 自动执行小游戏第一个开发任务

**方向：** 真实项目验证
**内容：**
- 创建小游戏第一个开发任务
- 用 run-next 自动执行
- 端到端验证

**依赖：** T019（验证项目就绪）
**复杂度：** 中

### T021 Reviewer Agent 自动审查 MVP

**方向：** 智能 Agent
**内容：**
- 读取验收标准和开发输出
- 生成审查结论
- 写入 reports/review/

**依赖：** T014（Agent 输出协议）、T016（模型适配器）
**复杂度：** 高

### T022 第二阶段阶段总结

**方向：** 总结
**内容：**
- 生成阶段总结报告
- 更新文档和经验沉淀
- 规划第三阶段

**依赖：** T013-T021 全部完成
**复杂度：** 低

## 5. 推荐执行顺序

```
T013（工作流协议） ─┐
T014（Agent 输出协议）├─ 协议设计阶段
T015（需求输入格式） ─┘
        ↓
T016（模型适配器）
        ↓
T017（Planner Agent）    ←── 依赖 T015 + T016
        ↓
T018（Main Agent 决策）  ←── 依赖 T014
        ↓
T019（游戏项目初始化）   ←── 依赖 T015
        ↓
T020（自动执行游戏任务）  ←── 依赖 T019
        ↓
T021（Reviewer Agent）   ←── 依赖 T014 + T016
        ↓
T022（阶段总结）
```

串行关键路径：T013/T014/T015 → T016 → T017 → T018 → T019 → T020 → T021 → T022

注意：实际执行中 runner.py 会按 tasks.md 中的顺序逐个执行，建议按以上顺序追加任务。

## 6. 暂不做的事情

1. **不做 Web UI** — 当前阶段保持 CLI 驱动
2. **不做数据库** — 继续使用文件系统存储
3. **不做多模型支持** — 只先支持一个国内模型
4. **不做完整游戏** — 小游戏只作为验证项目，不追求游戏完整性
5. **不做 Docker / 部署** — 专注核心自动化能力
6. **不做并发执行** — 保持串行执行，简化调试
7. **不做用户认证 / 权限** — 单用户 CLI 工具
8. **不做日志系统升级** — 继续使用文件日志

## 7. 风险与注意事项

1. **模型 API 稳定性** — 国内模型 API 可能有限额、超时、格式变更，适配器要做好错误处理
2. **Planner 生成质量** — 模型生成的任务清单可能质量不稳定，必须保留人工审核环节
3. **协议设计过度** — 协议不要设计得太复杂，够用即可，后续可以迭代
4. **验证项目范围膨胀** — 小游戏需求要克制，避免变成一个真正的游戏开发项目
5. **与第一阶段兼容** — 新功能不能破坏已验证的 run-next / run-loop 流程

## 8. 验收标准

第二阶段整体验收标准：

- [ ] 工作流、Agent 输出、需求输入的协议文档完成
- [ ] 国内模型 API 适配器可用

- [ ] Main Agent 决策规则已定义
- [ ] 小游戏验证项目需求就绪
- [ ] 用 runner 成功自动执行一个小游戏开发任务
- [ ] Reviewer Agent 能生成审查结论
- [ ] 第二阶段总结报告完成
